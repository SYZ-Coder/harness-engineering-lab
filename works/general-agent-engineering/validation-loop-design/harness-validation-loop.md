# Harness Engineering 的验证闭环：lint、测试、review、eval 怎么组合

> 面向这样的问题：知道要“加反馈回路”了，但验证到底怎么设计？  
> lint、单元测试、集成测试、人工 review、AI review、LLM-as-judge，它们分别管什么？顺序怎么排？怎么避免既低效又失真？

这篇文章的目标，是把“验证闭环”从一句口号拆成一套可执行的工程设计。

---

## 一、先说结论：验证闭环不是“多加几层检查”，而是把不同类型的错误分流到最合适的传感器

很多团队一开始理解“反馈回路”，会变成一种很朴素的堆叠：

- 加 lint
- 加测试
- 再加 code review
- 再加 AI review
- 再加 eval

结果常常是：

- 检查很多，但真正的问题没抓住
- 成本很高，但质量提升有限
- 同一个问题被重复检查好几遍
- 团队开始觉得这些流程都在“拖慢开发”

原因在于，验证闭环的重点并不是“层数多”，而是：

> **让不同类型的错误，在最便宜、最稳定、最靠前的位置被发现。**

这就意味着，验证体系不能只是“把所有检查都打开”，而要做传感器分工。

一个有效的验证闭环，通常要回答四个问题：

1. 哪类错误应该最早被发现？
2. 哪类错误应该由确定性工具判断？
3. 哪类错误必须引入语义评估或人类判断？
4. 哪类检查必须作为门控，哪类只做信号参考？

如果这四个问题没想清楚，验证越多，摩擦可能越大。

---

## 二、先分清楚：不同验证工具，抓的是不同类型的问题

验证闭环最重要的第一步，不是上工具，而是明确每类工具的职责边界。

## 1. lint / formatter / 静态规则

最适合抓：

- 代码风格
- 语法问题
- import 顺序
- 明确禁用 API
- 简单架构越界
- 常规安全规则

特点：

- 快
- 稳
- 便宜
- 适合每次都跑

不适合抓：

- 语义是否合理
- 功能是否真正正确
- 用户体验是否合格

## 2. 类型检查 / 编译检查

最适合抓：

- 类型不匹配
- 接口签名错误
- 缺失依赖
- 基本结构不一致

特点：

- 确定性强
- 对强类型代码库价值极高
- 是 harnessability 的天然优势来源

不适合抓：

- 运行时真实行为
- 需求理解错误

## 3. 单元测试

最适合抓：

- 核心逻辑错误
- 边界条件
- 关键函数的输入输出行为

特点：

- 快于集成测试
- 对回归防护很有价值
- 适合高频执行

局限：

- 容易“测试通过但设计错了”
- 如果测试本身由 AI 自动生成且未审查，可能只是复制实现思路

## 4. 集成测试 / 端到端测试

最适合抓：

- 模块协作问题
- 接口契约不一致
- 真实运行路径上的错误
- UI 交互流程问题

特点：

- 更接近真实使用
- 更能兜住跨模块问题

局限：

- 慢
- 维护成本高
- 不适合每次局部迭代都全量跑

## 5. 人工 code review

最适合抓：

- 设计方向错误
- 业务理解偏差
- 不必要复杂度
- 架构权衡问题
- 风险判断

特点：

- 适合高价值判断
- 能补足机械工具无法理解的上下文

局限：

- 昂贵
- 主观
- 容量有限

## 6. AI review / LLM-as-judge

最适合抓：

- 语义层面的模式问题
- 风格偏离
- 测试遗漏提示
- 文档与实现不一致
- UI 或文本质量的粗粒度评估

特点：

- 覆盖面广
- 可以做大规模初筛
- 能减少人工 review 的重复负担

局限：

- 不稳定
- 容易过度自信
- 需要校准
- 不能默认作为唯一真相

理解这一点之后，验证闭环的设计思路就会变得清晰很多：

> **不同层不是重复，而是分工。**

---

## 三、一个实用的分层模型：从便宜到昂贵，从确定到语义

如果把验证闭环按“成本与确定性”排开，一个比较实用的顺序是：

### 第 1 层：本地快速传感器

- formatter
- lint
- 类型检查
- 少量高价值单元测试

目标：

- 在最早阶段挡住明显错误
- 让错误尽可能在智能体仍可自我修正时暴露

### 第 2 层：任务相关验证

- 受影响模块的单元测试
- 关键集成测试
- 局部构建
- 结构测试

目标：

- 验证当前变更是否影响真实行为
- 避免每次都跑全量，拖慢反馈速度

### 第 3 层：高价值语义审查

- AI review
- 重点人工 review
- LLM judge
- UI / 文案 / 业务语义评估

目标：

- 抓住“技术上能跑但方向不对”的问题

### 第 4 层：上线前或高风险门控

- 全量测试
- 安全扫描
- 高风险改动人工批准
- 回归验证

目标：

- 把高代价错误挡在最后一道门外

这个分层的核心思想是：

> **越便宜、越确定的检查，越往前放；  
> 越昂贵、越语义化的检查，越聚焦地使用。**

这也是 Fowler 的 `Computational × Inferential` 框架在验证上的直接应用。

## 四、验证闭环真正要解决的，不只是“检测”，还有“自我纠偏”

很多人把验证理解成“发现问题”，但对 Harness Engineering 来说，还差一步：

> **发现问题之后，系统是否能基于信号纠偏。**

如果错误只是被报告给人，而没有形成后续动作，那它还不是完整闭环。

一个更完整的验证闭环通常是：

1. 变更产生
2. 传感器触发
3. 错误被分类
4. 智能体尝试修复或人类介入
5. 再次验证
6. 如果问题重复出现，把修复沉淀回 harness

这里最容易漏掉的是最后一步。

比如：

- AI 连续三次搞错 import 方式
- 你每次都手动改了

如果你没有：

- 补 lint 规则
- 补 AGENTS 提示
- 补示例

那这个系统并没有学习。

所以真正的验证闭环，最终要通向：

**反馈飞轮。**

## 五、不同验证层分别该承担什么门控责任

不是每个传感器都应该成为“硬门槛”。

这是验证设计里很关键的一点。

## 建议作为硬门控的

- formatter / lint
- 类型检查
- 核心单元测试
- 关键集成测试
- 高风险变更确认

为什么：

- 判断标准清楚
- 自动化程度高
- 成本可控
- 误伤率相对低

## 适合作为软信号的

- AI code review
- LLM-as-judge
- 风格一致性语义评分
- 草稿级别的 UI 评估

为什么：

- 这类信号往往概率性更强
- 更适合提示“这里值得再看一眼”
- 不适合直接充当唯一裁决者

## 可以升级为硬门槛的例外情况

当某个语义性传感器经过足够校准、而且任务模式很固定时，它也可能承担更强门控责任。

例如：

- 表单提交流程是否完整
- 页面是否存在明显空态缺失
- 某类客服回复是否违反硬性政策

但这种升级应该建立在长期数据和校准基础上，而不是一开始就默认相信。

## 六、一个适合中小团队的最小验证闭环

如果团队刚开始引入 Harness Engineering，不要先做一个巨大的验证系统。

更建议从下面这套最小闭环开始。

## 本地层

每次智能体完成改动后，至少跑：

- formatter
- lint
- 类型检查
- 与当前任务直接相关的测试

## 提交层

在 PR 或合并前，至少补：

- 受影响模块的关键测试
- 一次 AI review 或 checklist review
- 高风险改动的人工确认

## 高风险层

如果改动涉及：

- 数据库 schema
- 鉴权
- 支付
- 部署
- 跨模块公共接口

则增加：

- 人工 review
- 更完整的回归测试
- 变更说明或 ADR 更新

这个版本已经足够覆盖大多数中小团队最常见的风险，而不至于流程重到无法使用。

## 七、一个更成熟的闭环：按错误类型分流

下面给一个更成熟但仍然可实施的设计思路。

### 类型 A：格式和机械错误

传感器：

- formatter
- lint
- 类型检查

处理方式：

- 直接硬门控
- 智能体自行修复

### 类型 B：局部逻辑错误

传感器：

- 单元测试
- 局部集成测试

处理方式：

- 硬门控
- 智能体先修，失败再人工介入

### 类型 C：架构和边界问题

传感器：

- 结构测试
- 依赖规则
- 人工 review

处理方式：

- 重要规则设为硬门控
- 复杂权衡交由人工判断

### 类型 D：语义和体验问题

传感器：

- AI review
- LLM judge
- 人工体验审查
- Playwright 类真实交互验证

处理方式：

- 先做软信号
- 高风险或关键页面再升级为强门控

### 类型 E：长期漂移问题

传感器：

- 死代码扫描
- 覆盖率趋势
- 质量评分趋势
- 周期性 review

处理方式：

- 不一定阻塞当前提交
- 但应进入后续垃圾回收或重构任务

这个分流模型的好处是，你不会把所有问题都塞给同一种传感器。

## 八、为什么很多团队验证做了很多，效果还是不好

因为最常见的问题，不是“缺少验证”，而是“验证配置错误”。

## 问题一：把 AI review 当最终裁判

后果：

- 错误自信
- 误判率高
- 解释看起来很合理，但结论不一定对

正确方式：

- 让 AI review 做初筛
- 让人或确定性规则做关键裁决

## 问题二：每次都跑全量测试

后果：

- 反馈太慢
- 智能体修复节奏被拖垮
- 团队逐渐跳过验证

正确方式：

- 本地跑局部
- 关键路径跑必需
- 合并前再跑全量

## 问题三：只检查“有没有跑通”，不检查“方向对不对”

后果：

- 技术正确，产品错误
- 测试全绿，但用户体验很差

正确方式：

- 补充语义型传感器
- 让人工集中审查高价值方向问题

## 问题四：发现错误但不沉淀规则

后果：

- 同样的错误反复发生
- 验证系统没有学习能力

正确方式：

- 把重复问题写回 AGENTS、规则、测试或 lint

## 九、如何把验证闭环接进团队工作流

验证闭环只有接进真实工作流才有意义。

一个比较自然的接法是：

### 任务开始前

- 明确完成标准
- 明确哪些检查必须通过
- 明确哪些改动属于高风险

### 智能体执行时

- 先跑便宜传感器
- 有失败就先自修
- 降低明显错误流入下一层的概率

### 提交前

- 跑任务相关验证
- 做 AI review 或 checklist review
- 对高风险改动做人类审查

### 合并后

- 收集失败模式
- 复盘哪些错误本可更早发现
- 更新 AGENTS、规则、hooks、测试

这时候，验证就不再是一组孤立命令，而是团队的持续学习系统。

## 十、一个推荐的最小组合

如果你现在就想搭最小验证闭环，我会推荐这个组合：

### 必须有

- lint
- 类型检查或编译检查
- 核心单元测试
- 高风险改动确认机制

### 建议有

- 任务相关集成测试
- AI review 作为初筛
- 关键路径人工 review

### 以后再加

- LLM judge
- UI 自动评估
- 质量趋势跟踪
- 垃圾回收智能体

这套顺序的重点是：

**先建立稳定闭环，再逐步增加高级传感器。**

---

## 十一、一个判断标准：你的验证闭环是否健康

你可以用下面几个问题自查：

1. 明显错误是否能在最早阶段暴露？
2. 是否存在大量重复、低价值的检查？
3. 高价值的人类判断是否被留给真正需要它的问题？
4. AI review 是否只是辅助，而不是唯一裁判？
5. 重复错误是否会反向沉淀到 harness？

如果前两个问题答“否”，说明闭环太弱。  
如果后三个问题答“否”，说明闭环虽然存在，但还没有真正形成系统性学习。

---

## 总结

Harness Engineering 的验证闭环，不是“检查越多越好”，而是：

- 把错误按类型分流
- 把检查按成本和确定性分层
- 把门控和信号区别开
- 把发现问题后的修复沉淀回系统

真正高质量的验证闭环，最终会形成这样一条链路：

> **规则引导行动 -> 传感器发现偏差 -> 系统或人纠偏 -> 经验写回 harness**

这时候，验证不再只是防线，而会变成 Harness Engineering 最有价值的学习机制。
