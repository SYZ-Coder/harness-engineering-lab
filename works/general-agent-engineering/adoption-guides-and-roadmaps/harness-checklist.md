# Harness Engineering 落地清单：一页式检查表

> 面向这样的问题：概念看完了，文章也读了，但如果今天就开始做，到底该检查哪些项？
>
> 这份清单不追求全面覆盖所有高级玩法，而是帮助你判断：一个团队或个人项目，是否已经具备最小可用的 Harness Engineering 落地条件。

---

## 一、最小入口

- [ ] 仓库里有一份可维护的 `AGENTS.md`
- [ ] `AGENTS.md` 说明了项目结构和关键目录职责
- [ ] `AGENTS.md` 说明了当前任务应先读什么
- [ ] `AGENTS.md` 明确了禁止事项和高风险区域
- [ ] `AGENTS.md` 说明了最小验证命令

如果这几项还没有，通常说明你还没有稳定入口。

---

## 二、规则显式化

- [ ] 团队关键约定已经不只存在于口头沟通中
- [ ] 至少有一部分架构规则写进了仓库文档
- [ ] 关键术语、边界、目录职责有固定说明
- [ ] 高风险变更类型已被显式列出
- [ ] 已经能区分“建议”与“硬约束”

如果规则还主要靠人记忆，智能体就只能靠猜。

---

## 三、验证闭环

- [ ] 已接入 lint 或等价静态检查
- [ ] 已接入类型检查或编译检查
- [ ] 核心逻辑有基础测试
- [ ] 高风险改动需要额外确认或人工 review
- [ ] 团队能区分硬门控与软信号

如果没有验证闭环，Harness 很容易退化成“更详细的提示词”。

---

## 四、反馈飞轮

- [ ] 团队会记录智能体的高频失败模式
- [ ] 发现重复错误后，会回写到规则、测试或工具中
- [ ] 团队会定期复盘哪些错误本可更早发现
- [ ] 某些规范已经从文字升级为 hooks、lint 或结构测试
- [ ] 已经形成“发现问题 -> 修正 harness -> 观察复发”的节奏

如果错误每次都靠人工临时修复，系统还没有真正开始学习。

---

## 五、任务与风险控制

- [ ] 团队不会一开始就把高风险任务交给智能体
- [ ] 任务边界通常是清楚的、可验收的
- [ ] 团队知道哪些任务适合 AI，哪些不适合
- [ ] 会限制单次改动范围和无关重构
- [ ] 危险操作有确认机制

Harness 的目标不是“让 AI 接管一切”，而是“让它在可控范围内稳定做对事”。

---

## 六、架构与复杂度

- [ ] 团队已经先跑通最小单智能体闭环
- [ ] 只有在真实需要时，才考虑循环式或多智能体
- [ ] 团队不会把“更复杂的编排”误当成“更成熟的能力”
- [ ] 如果考虑平台化，已经有跨团队复用和治理需求
- [ ] 团队会随着模型升级重新评估现有 harness

复杂度应该跟着真实问题增长，而不是跟着热词增长。

---

## 七、快速判断

### 如果你现在满足下面这些项

- 有 `AGENTS.md`
- 有 lint / test / typecheck
- 有高风险确认机制
- 会回写重复问题

那么你已经有了最小可用的 Harness Engineering 雏形。

### 如果你还缺下面这些项

- 没有稳定入口
- 没有验证闭环
- 没有失败复盘
- 规则还主要靠口头传递

那么最值得优先做的，不是上多智能体或平台，而是先把基础闭环补齐。

## 总结

如果把 Harness Engineering 理解成一句最实用的话，那就是：

> **先有入口，再有约束，再有验证，再有反馈飞轮。**

这份清单的价值，不在于“全部打钩”，而在于帮你看清：  
你现在到底缺的是规则、验证、反馈，还是只是顺序错了。
