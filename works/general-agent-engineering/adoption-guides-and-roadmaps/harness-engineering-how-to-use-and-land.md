# Harness Engineering 到底怎么用：从概念到落地实施指南

> 面向这样一类问题：网上很多人在讲“驾驭工程”“Harness Engineering”，但真正关键的问题往往没有说透——它到底怎么用？如何落地？怎么实施？适合谁？从哪一步开始？
>
> 本文基于 OpenAI 官方文章、菜鸟教程的概念介绍，以及本仓库已整理的 Ralph、Anthropic、LangChain、Martin Fowler 等资料进行综合梳理，目标不是重复定义，而是给出一个能指导实践的回答。

---

## 一、先说结论：Harness Engineering 不是“更强的提示词”，而是 AI 时代的工程控制系统

很多人第一次听到 Harness Engineering，会误以为它只是 Prompt Engineering 的升级版，或者是“多加几个规则文件”。这其实抓错了重点。

更准确的理解是：

> **Harness Engineering = 为智能体设计环境、约束、反馈回路和验收机制，让它在可控轨道上持续完成工作。**

如果说传统软件工程的基本范式是：

```text
人写代码 -> 机器执行代码
```

那么 Harness Engineering 对应的范式更接近：

```text
人设计约束与反馈系统 -> 智能体生成与修改代码 -> 机器执行 -> 传感器反馈 -> 智能体或人修正
```

OpenAI 在 2026 年提出这个概念，本质上是在说：当“写代码”这件事越来越多由智能体承担后，工程师的核心产出正在从“代码本身”转向“约束系统”。

因此，Harness Engineering 的重点不在于：

- 怎么把提示词写得更像魔法咒语
- 怎么让 AI 一次生成 1000 行代码
- 怎么追求完全自动化

而在于：

- 怎么让 AI 在正确的边界内工作
- 怎么让错误被尽可能早地发现
- 怎么让系统可以自我纠偏，而不是靠人类事后救火
- 怎么把团队原本靠经验维持的秩序，显式化为机器可执行的约束

这也是为什么 OpenAI、Anthropic、LangChain、Martin Fowler 这几条脉络最后都走向了同一个方向：

**真正难的已经不再是“让模型会写代码”，而是“如何设计围绕模型的外层系统”。**

---

## 二、什么是 Harness：它不是一个工具，而是一整层“模型之外的控制层”

LangChain 对这个概念给出了一个非常重要的定义：

> **Agent = Model + Harness**

也就是说，裸模型不是智能体。模型本身只会接收输入、输出文本。它并不天然具备以下能力：

- 持久状态
- 访问文件系统
- 执行命令
- 浏览网页
- 读取实时知识
- 理解你团队的代码规范
- 在失败后自动重试并纠偏
- 判断什么叫“完成”

这些能力都来自 harness。

因此，所谓 harness，可以理解为：

> **模型之外的一切代码、配置、约束、执行逻辑、反馈机制和运行环境。**

一个完整的 harness，通常包含下面这些组成部分：

### 1. 指令与知识入口

- `AGENTS.md`
- 架构说明
- 术语表
- ADR / 决策记录
- 技术约束说明
- 团队标准文档

### 2. 工具层

- 编辑器
- 终端
- 浏览器
- MCP
- 数据库工具
- API 调用能力

### 3. 执行环境

- 文件系统
- 沙箱
- 容器
- 测试环境
- 运行时依赖

### 4. 编排层

- 单智能体循环
- 子智能体分工
- 任务拆分
- handoff 机制
- 上下文切换与续接

### 5. 反馈层

- lint
- 类型检查
- 单元测试
- 结构测试
- 架构约束检查
- AI code review
- LLM-as-judge
- 人工 review

### 6. 记忆层

- 仓库中的文档和文件
- scratchpad
- session 事件流
- memory folder
- 任务日志
- 执行轨迹

### 7. 门控层

- 失败禁止继续
- 危险操作需确认
- 质量评分不达标不能合并
- 没有完成信号就不退出

从这个角度看，Harness Engineering 其实就是：

> **把原本隐性的“人类经验、团队规范、工程秩序、验证习惯”，逐步编码成可被智能体消费、可被系统执行、可被反馈修正的外层控制结构。**

## 三、Harness Engineering 到底靠什么机制起作用

如果只问“它有什么组件”，其实还不够。更重要的问题是：这些组件如何协同工作？

综合 OpenAI、Fowler、Anthropic、LangChain 的材料，可以把 Harness Engineering 的工作机制概括成四个核心循环。

## 机制一：前馈引导

Fowler 用 `Guides` 这个词来概括“行动前的引导器”。

这类机制的目标不是在错误发生后抓住它，而是在它发生前尽量减少概率。

典型形式包括：

- `AGENTS.md` 告诉智能体项目怎么导航
- `architecture.md` 告诉它哪些边界不能越过
- 技能包告诉它某种任务的正确做法
- 脚手架模板让它默认生成符合结构的代码
- 示例文件让它模仿项目已有风格

前馈的价值在于提高“首次成功率”。

没有前馈时，智能体会依赖训练集中最常见、最平均的做法，而这往往和你项目的真实需求并不完全一致。于是它会输出“通用正确、局部错误”的结果。

这也是为什么 OpenAI 和 HumanLayer 都强调：

- 不要把关键规则只放在人脑里
- 不要写巨型手册
- 要做“地图而不是百科全书”
- 要渐进式披露，而不是一次把所有规则塞满上下文

## 机制二：反馈传感

仅有前馈是不够的。因为再好的规则，也无法覆盖所有现实情况。

所以 Harness 必须有 `Sensors`，也就是“行动后的观测器”。

这类机制包括：

- lint
- 类型检查
- 单元测试
- 集成测试
- 结构测试
- 架构 fitness functions
- AI code review
- LLM-as-judge
- 人工 review

反馈机制的作用，是在结果偏离预期时发出信号，让系统纠偏。

Fowler 特别强调了一个重要观点：

> 只有反馈没有前馈，智能体会反复犯类似错误。  
> 只有前馈没有反馈，你不知道规则是否真的生效。

所以 Harness 本质上不是“规则堆叠”，而是**前馈与反馈协同的控制闭环**。

## 机制三：机械化执行与背压门控

OpenAI 文章里一个非常关键的思想是：**不要依赖智能体自我感觉良好地宣布完成。**

这就引出了 Harness 的第三类机制：机械化执行和背压。

所谓背压，可以理解为：

> 不规定你必须怎么做，但如果结果不符合条件，你就不能继续。

比如：

- 测试没过，不能进入下一轮
- 架构规则违反，不能提交
- 风险操作需要人工确认
- 质量评分不达标，不能合并

这类机制的好处非常明显：

- 它不压制智能体的探索空间
- 但它能拦住明显坏结果
- 它让“完成”变成可检查条件，而不是主观判断

这也是 Ralph、Anthropic、OpenAI 这些实践都特别强调验收门控的原因。

## 机制四：上下文治理与外部记忆

大模型在长任务中的一个天然问题，是上下文会腐化：信息变多、指令过期、历史噪声上升、模型开始“差不多就行”。

LangChain 把这个问题叫做 `context rot`，Anthropic 也谈到 `context anxiety`。

所以 Harness 必须解决两个问题：

1. 重要信息怎么保留
2. 无关噪声怎么清理

常见做法包括：

- 把任务和决策写入仓库文件
- 用 scratchpad 记录中间状态
- 用 memory folder 保存跨会话经验
- 用 compaction 做上下文压缩
- 用子智能体做上下文隔离
- 用 context reset 避免长对话衰退

这一点很重要：  
**Harness Engineering 的记忆，不是“让模型全记住”，而是“把该记的东西放到模型可重新读取、可持续维护的位置上”。**

## 四、OpenAI 说的六大概念，分别落到哪里

OpenAI 原文提出了六个核心概念。很多人知道这些名词，但不知道它们在工程上分别对应什么。下面用“概念 -> 工程含义 -> 实施动作”的方式重新梳理。

### 1. 仓库即记录系统

工程含义：

- 对智能体来说，不在仓库里的东西基本等于不存在
- 脑子里的约定、Slack 里的讨论、口头共识，都不是稳定输入

实施动作：

- 把关键约束写成 `AGENTS.md`
- 把设计决策写成 ADR 或 `DECISIONS.md`
- 把任务描述写成文件，而不是只在聊天窗口里说
- 把术语、目录职责、边界说明放进仓库

### 2. 地图而非手册

工程含义：

- 入口文档要短、清楚、可导航
- 不要试图用一个超长文档解决所有问题

实施动作：

- `AGENTS.md` 保持短小
- 用目录导航引导智能体逐层读取
- 把详细规则下沉到专门文档
- 用渐进式披露降低上下文噪声

### 3. 机械化执行

工程含义：

- 规则不能只写给人看，还要尽可能变成机器能执行的检查

实施动作：

- 先自然语言规则
- 再示例化
- 再变成 lint / hooks / structure tests / CI 校验

### 4. 智能体可读性

工程含义：

- 代码库、技术栈、目录结构、命名方式，本身会影响智能体成功率
- 越清晰、越常见、越结构化，越容易被 harness

实施动作：

- 使用成熟框架和常规结构
- 提高模块边界清晰度
- 避免过度花哨但无收益的写法
- 用统一命名和一致目录降低歧义

### 5. 吞吐量改变合并理念

工程含义：

- 智能体能产生大量改动，速度可能不再是瓶颈
- 真正的瓶颈会转移到 review、验证、合并和治理

实施动作：

- 改造 review 与门控机制
- 关注 WIP 和并发限制
- 提升验证自动化，而不是单纯增加生成速度

### 6. 熵管理

工程含义：

- 智能体会复制仓库里已有模式，包括坏模式
- 如果不主动治理，代码库会加速漂移

实施动作：

- 定期做重构与规则升级
- 记录重复失败模式
- 建立反馈飞轮
- 用“垃圾回收”机制扫描偏差与技术债

这六个概念的共同点是：  
**它们都不是单点技巧，而是在重新定义“工程师如何控制系统”。**

## 五、Fowler 给出的关键升级：Harness 是控制论系统，不是规则文件合集

如果说 OpenAI 给出了“现象与实践感”，那么 Martin Fowler / Birgitta Böckeler 给出的贡献，是把 Harness Engineering 从经验技巧提升成一个更稳定的分析框架。

她最重要的框架是：

```text
Guides × Sensors
Computational × Inferential
```

也就是把所有 harness 组件放进一个 2×2 矩阵里。

|  | 计算性（确定性） | 推理性（语义性） |
|--|--|--|
| 前馈引导 | 脚手架、OpenRewrite、模板、LSP | AGENTS.md、Skills、架构说明 |
| 反馈传感 | lint、类型检查、结构测试、覆盖率 | AI review、LLM judge、语义评估 |

这个框架有三个非常重要的现实含义。

### 含义一：不是所有检查都值得用 LLM 做

能用确定性工具解决的，就优先用确定性工具。

因为：

- 更快
- 更便宜
- 更稳定
- 更容易放进 CI 和 hooks

例如：

- import 顺序
- 命名规则
- 目录边界
- 类型问题
- 依赖禁用关系

这些都不应该主要靠 LLM review。

### 含义二：也不是所有问题都能靠 lint 解决

有些问题本质上是语义性的，必须引入推理型传感器，例如：

- 这个页面有没有明显 AI 模板味
- 这个 API 设计是否违背了真实业务意图
- 这段实现是不是“技术上可运行，但方向错了”
- 这个测试是否只验证了表面行为

这些问题往往不能只靠确定性工具捕捉。

### 含义三：Harness 的重点不只是维护性，还包括架构与行为

Fowler 把 harness 分成三个调控维度：

- 可维护性 Harness
- 架构适应度 Harness
- 行为 Harness

其中：

- 可维护性最成熟
- 架构适应度次之
- 行为 Harness 最弱

这点特别值得重视，因为很多人谈 Harness Engineering 时，容易把它等同于“加 lint、加测试、加规则”。但真正难的地方其实在行为验证：

> 功能真的对了吗？  
> 设计真的符合需求吗？  
> 用户体验真的可接受吗？

这也是为什么 Anthropic 要引入独立 Evaluator，LangChain 要强调 eval readiness，Fowler 会说“行为 harness 是房间里的大象”。

## 六、常见架构设计有哪些：不是只有一种 Harness

很多人在了解 Harness Engineering 后会立刻问：

> “那标准架构是什么？”

答案是：没有唯一标准架构，但有几类典型形态。

## 方案一：最小单智能体 Harness

这是绝大多数团队最应该从这里开始的方案。

### 架构

- 一个主智能体
- 一个简洁的 `AGENTS.md`
- 常用工具：终端、编辑器、搜索、浏览器
- 一套基础验证：lint / test / typecheck
- 必要的人工 review

### 优点

- 成本最低
- 最容易落地
- 维护最简单
- 对模型能力依赖较低

### 适用场景

- 个人项目
- 小团队
- 新功能开发
- 测试补全
- 局部重构

### 风险

- 单智能体容易“自评过高”
- 长任务中容易上下文退化
- 如果缺少反馈回路，容易沦为 vibe coding

## 方案二：循环式 Harness

Ralph 是这类方案的代表。

### 架构

- 任务写入文件
- 通过循环不断运行智能体
- 每轮在新上下文中继续
- 用 scratchpad 记录状态
- 明确完成信号

### 常见角色

- Planner
- Builder
- Critic
- Finalizer

### 优点

- 更适合长任务
- 自愈能力更强
- 对上下文腐化更有韧性
- 任务完成条件更清晰

### 风险

- 编排复杂度上升
- 回合太多会产生额外成本
- 如果验收设计不好，会反复空转

## 方案三：多智能体分工 Harness

Anthropic 的长时应用开发就是这类设计。

### 架构

- Planner 负责把模糊目标转为高层规格
- Generator 负责实现
- Evaluator 负责独立验证

### 关键机制

- sprint 合同
- done 标准协商
- 独立评审和打分
- 通过真实交互进行验证

### 优点

- 对复杂任务更稳
- 能缓解自我评估偏差
- 适合高不确定性任务

### 风险

- 成本高
- 设计不当会过度工程化
- 模型升级后旧结构可能迅速过时

## 方案四：平台化 / Meta-Harness

这是更大规模组织会关心的方向。

### 架构

- Session 持久化
- Harness 无状态
- Sandbox 独立 provision
- 工具接口标准化
- 安全边界和凭证隔离

### 优点

- 可替换性强
- 可观测性更好
- 更适合多租户、平台团队、企业级治理

### 风险

- 基础设施复杂
- 初期投入高
- 不适合大多数团队一开始就做

真正重要的不是“选最先进的”，而是：

> **用与你当前问题复杂度相称的 harness。**

Anthropic 的经验特别值得记住：

**每个 harness 组件都编码了一个假设。**  
如果模型升级后那个假设不再成立，这个组件就可能变成负担。

所以 Harness 不应默认越复杂越好，而应该默认：

**从最简单开始，按需增加复杂度。**

## 七、到底怎么落地：一个团队可以直接照着做的实施路径

下面给出一个相对稳妥的落地顺序。它的目标不是“一步到位搭平台”，而是先建立最小可行 harness，然后逐步演化。

## 第一步：选一个边界清楚的试点

不要上来就说“以后所有开发都交给 AI”。

更好的做法是，先选一个：

- 新功能模块
- 非核心子系统
- 文档或脚手架工程
- 测试补全任务
- 独立重构专题

试点范围必须满足两个条件：

- 边界相对清楚
- 能建立明确的完成标准

如果范围太模糊，你根本无法判断 harness 是否有效。

## 第二步：建立最小入口文件

至少要有一份像样的 `AGENTS.md`，内容不必大而全，但必须解决以下问题：

- 这个项目怎么导航
- 关键目录分别负责什么
- 哪些约定必须遵守
- 哪些行为禁止出现
- 当前任务的阅读顺序是什么

这里最容易犯的错误，是写成百科全书。  
正确方式是写成地图，让智能体知道“先看什么，再去哪儿找详细规则”。

## 第三步：把“完成”定义为可检查条件

这是 Harness 能否落地的分水岭。

你必须明确：

- 测试通过算不算完成
- 是否还要求 lint、类型检查、覆盖率
- 是否要求结构约束不过界
- 是否要人工 review
- 哪些行为必须用户确认

如果没有这些条件，智能体的“完成”就只能是一种主观感受。

## 第四步：先把自然语言规则写出来

很多团队一听“机械化执行”，就想立刻做复杂 lint。

其实顺序应该是：

1. 先把规则写成自然语言
2. 再补示例
3. 再看哪些规则值得自动化

因为不是每个规范都值得工程化。

最值得优先自动化的，通常是：

- 高频违反
- 影响大
- 判断标准清楚
- 可被机器稳定识别

## 第五步：把反馈回路接起来

真实落地时，你需要建立一个最小反馈飞轮：

1. 发现智能体反复犯某类错
2. 分析根因
3. 决定修 prompt、修工具、修规则还是修验证
4. 把修复写回 harness
5. 观察后续是否复发

如果只是“这次改对了就算了”，你得到的不是 harness，而是一次次临时救火。

## 第六步：控制并发与吞吐量

智能体往往能极快地产生大量代码，但真正限制交付的通常不是“写得慢”，而是：

- 评审跟不上
- 验证跟不上
- 合并跟不上
- 质量漂移变快

因此，早期落地时需要刻意控制：

- 同时跑多少任务
- 每个 PR 大小
- 每轮变更范围
- review 节奏

先建立闭环，再扩大吞吐量。

## 第七步：定期做 Harness 瘦身

Harness 不是一次性资产，而是持续演化的系统。

你需要周期性检查：

- 哪些规则已经形同虚设
- 哪些补偿机制是旧模型时代遗留
- 哪些工具反而增加噪声
- 哪些传感器从不触发，是真的没问题还是根本没覆盖到

这一步非常关键，因为不做瘦身，harness 自身也会变成负担。

## 八、对不同团队，有哪些不同落地方案

Harness Engineering 不适合用一套方法打天下。不同规模、不同成熟度的团队，应该采用不同版本。

## 方案 A：个人开发者最小版

适合：

- 独立开发者
- 副业项目
- 小型开源项目

建议配置：

- 一个 `AGENTS.md`
- 一套基础测试和 lint
- 关键设计自己做
- AI 负责实现、补测试、做重构草稿
- 每批生成后人工通读
- 把重复错误写回规则

这个版本的核心不是“自动化程度高”，而是：

**自己必须保有设计主导权。**

仓库里的 `thinking/harness-for-solo-developers.md` 已经说明了这一点：  
个人开发者没有团队兜底，所以更需要 harness，但不需要重型平台。

## 方案 B：中小团队标准版

适合：

- 3 到 10 人团队
- 产品研发组
- 平台工程起步阶段

建议配置：

- 项目级 `AGENTS.md`
- 关键任务模板
- lint / test / typecheck / hooks
- ADR / 架构说明
- 限制危险操作
- 重点模块引入结构测试
- 关键任务做人工 review

这个阶段最重要的是：

- 把团队标准编码出来
- 建立反馈飞轮
- 减少 review 摩擦

## 方案 C：多智能体增强版

适合：

- 长任务
- UI / 全栈一体任务
- 需要独立评估的复杂工作

建议配置：

- Planner / Builder / Evaluator 分工
- 明确的 sprint 合同
- 自动化交互验证
- 结果评分与 critique

注意：  
这个版本只有在任务确实复杂、单智能体经常自评失真时才值得使用。

## 方案 D：平台化企业版

适合：

- 多团队共享平台
- 企业级合规与安全需求
- 多租户、长期运行、跨会话场景

建议配置：

- session 持久化
- sandbox 标准化
- harness 可替换
- 审计与可观测性
- 令牌与凭证结构性隔离

这一层已经不只是“开发工具优化”，而是平台工程。

## 九、已经有哪些“落地案例”，它们分别是怎么落地的

这一部分回答“有没有真实世界已经这么做了”的问题。

答案是：有，而且不止一种形态。

## 案例一：OpenAI Codex 团队

这是 Harness Engineering 这个词被正式提出的起点。

### 他们怎么落地

- 从空仓库开始
- 采用零手写代码的刻意约束
- 所有规范、计划、任务都尽量写入仓库
- 依赖自定义 linter、结构测试、质量评分进行机械门控
- 通过“垃圾回收”机制处理长期漂移和技术债

### 给我们的启发

- 工程师的工作重心确实会从编码转向设计控制系统
- 智能体不是一次调用，而是长期生产单元
- 不建立反馈与约束，吞吐量只会放大问题

## 案例二：Ralph / Ralph Orchestrator

这是开源社区里非常具体、非常可复现的 harness 实践。

仓库里 `practice/01-ralph-demo/README.md` 已经做了实验记录。

### 它怎么落地

- 用文件定义任务
- 用 Planner / Builder / Critic / Finalizer 四类角色循环推进
- 用 scratchpad 保存跨轮状态
- 每轮在新上下文中继续
- 用明确 completion signal 终止循环

### 给我们的启发

- 长任务不一定要靠单轮超长上下文硬扛
- 新上下文 + 外部状态，是提高稳健性的可行办法
- “完成”最好是可检查信号，而不是模型主观输出

## 案例三：Anthropic 长时应用开发 Harness

这是目前“多智能体协同 + 独立验证”最清晰的工程案例之一。

### 它怎么落地

- Planner 把模糊提示扩成高层产品规格
- Generator 按 sprint 实现
- Evaluator 用 Playwright 实际跑应用验证
- Generator 与 Evaluator 先协商 sprint 完成标准

### 给我们的启发

- 模型很容易过早宣布完成
- 自评能力并不可靠
- 对复杂任务，引入独立 Evaluator 是有价值的
- 但这种结构是否值得，取决于任务是否处于模型能力边界

## 案例四：Anthropic Managed Agents

这是“平台化 harness”的真实落地。

### 它怎么落地

- 把 session、harness、sandbox 解耦
- 把凭证访问放在沙箱之外
- 让 harness 无状态化、可替换
- 通过标准接口调用工具与环境

### 给我们的启发

- 当规模变大时，harness 本身也需要像基础设施一样被治理
- 安全边界不能只靠 prompt 约束，要靠结构性隔离

## 案例五：LangChain / Deep Agents

这是“把 harness 当成持续优化对象”的实践。

### 它怎么落地

- 系统化拆分 prompts、tools、skills、memory、orchestration
- 用 traces 改进 harness
- 把学习分成模型层、harness 层、上下文层

### 给我们的启发

- 不要把问题都推给模型
- 很多性能提升，来自 harness 层而非权重层
- 上下文层和 harness 层的演化，比大多数人想的更值得投入

## 案例六：个人开发者的轻量落地

从 Lalit Maganti 以及本仓库的思考可以看出，个人开发者也能落地 Harness Engineering，只是形态不同。

### 它怎么落地

- 自己做设计和关键判断
- AI 做实现、补测试、重构草稿
- 每批代码后强制通读和重构
- 持续把重复错误写回规则与流程

### 给我们的启发

- 个人开发者不需要重型基础设施
- 但必须避免把自己变成“只会点头的项目经理”
- Harness 的本质，是守住判断权

## 十、为什么很多人看了概念，还是不会用

这是最现实的问题。

原因通常不是“概念太难”，而是下面这几个误区太常见。

## 误区一：把 Harness 当成规则文件

很多人以为加一个 `AGENTS.md` 就算做了 harness。

其实没有反馈、没有门控、没有记忆、没有验收，那只是“写了说明书”，不是控制系统。

## 误区二：把 Harness 当成多智能体编排

多智能体只是其中一种实现方式。

Harness 的本质是控制与反馈，不是角色数量。

一个单智能体系统，只要有明确边界、验证闭环和反馈飞轮，也是在做 Harness Engineering。

## 误区三：直接追求重型架构

很多团队刚开始就想做：

- session 层
- meta-harness
- 自动优化外循环
- 多代理平台

但连最基本的：

- 规则落盘
- 测试门控
- 反馈闭环

都还没建立。

这样最后大概率是“平台先复杂了，问题还没解决”。

## 误区四：以为模型升级就能替代 Harness

模型升级确实会让一部分 harness 组件变成死重，但不会让 harness 消失。

模型越强，越能在更大空间里行动；  
行动空间越大，越需要好的边界和反馈。

变化的是 harness 的形态，不是它的必要性。

## 误区五：把 Harness 理解成“为了完全消灭人工参与”

这是最危险的误区。

Fowler 说得很准确：

> 好的 harness，不一定要以完全消除人类输入为目标，而是要把人类输入引导到最重要的地方。

这意味着：

- 人不是被移除
- 人是被重新定位

真正成熟的 Harness，不是让人退出回路，而是让人只在高价值决策点介入。

## 十一、我认为最值得记住的 5 个落地原则

如果整篇文章只保留 5 条原则，我会留这 5 条。

### 原则 1：先把隐性知识写进仓库

不进仓库，就不稳定；  
不稳定，就不能成为可靠输入。

### 原则 2：先做最小 Harness，再做复杂编排

先建立最基本的约束和反馈闭环，再考虑多智能体、平台化和自动优化。

### 原则 3：优先把高频、高影响、可判断的规范机械化

不是所有规则都值得自动化。  
先抓最能降低摩擦的那部分。

### 原则 4：把“完成”定义成可验证结果，而不是模型口头承诺

没有验收门控，就没有真正的 harness。

### 原则 5：Harness 不是一次设计完成的，而是靠反馈飞轮持续演化

每次重复出现的问题，都是下一轮 harness 演进的输入。

## 十二、如果今天就开始，你可以怎么做

如果你现在就想把 Harness Engineering 用起来，而不是停留在理解层，我建议从下面这个最小版本开始：

### 个人版最小清单

1. 建一个 `AGENTS.md`
2. 写清项目结构、术语、禁止事项
3. 给核心逻辑补一套基础测试
4. 把 lint / typecheck 接进日常流程
5. 每次让 AI 改完代码后，自己通读关键部分
6. 记录重复错误，更新规则

### 团队版最小清单

1. 建立仓库级 `AGENTS.md`
2. 把团队标准分成自然语言规则和可执行规则
3. 接通 lint / test / hooks / CI
4. 给关键模块补结构约束
5. 限制危险操作与大范围改动
6. 建立失败复盘与反馈飞轮

### 不要一开始做的事

- 不要先做复杂多智能体平台
- 不要一开始就写超长规则大全
- 不要把 AI 的完成声明当成完成
- 不要把所有规范都一股脑自动化
- 不要忽略模型升级后对 harness 的再评估

## 十三、总结：Harness Engineering 真正改变的，不是写代码的速度，而是工程师控制系统的方式

所以，回到文章开头那个问题：

> 驾驭工程到底怎么用？如何落地？怎么实施？

最简洁的回答是：

**先把约束、反馈、记忆和验收机制搭起来，再让 AI 在这个系统里工作。**

它不是一个神奇工具，也不是某种固定框架，更不是“AI 自己会变好”的幻想。它是一种新的工程实践：

- 把隐性经验显式化
- 把主观判断尽量转为可执行约束
- 把智能体输出接入验证闭环
- 把反复犯错的地方沉淀回 harness
- 把人类从低价值重复劳动中释放出来，重新聚焦高价值判断

从这个意义上说，Harness Engineering 的真正落点不是“AI 替代工程师”，而是：

> **工程师开始像设计控制系统一样设计开发流程。**

这可能才是它最值得被认真对待的地方。
