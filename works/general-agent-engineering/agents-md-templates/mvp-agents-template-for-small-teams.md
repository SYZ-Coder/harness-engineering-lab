# 中小团队 AGENTS.md 最小模板：如何给编码智能体一张清晰地图

> 面向 3 到 10 人团队、个人项目升级阶段、以及刚开始引入编码智能体的仓库。
>
> 目标不是写出一份“最全规则大全”，而是做出一份真正有用、能长期维护、能被智能体稳定消费的最小入口文件。

---

## 一、先说结论：AGENTS.md 的价值不在“信息多”，而在“导航清楚”

很多团队第一次写 `AGENTS.md`，会不自觉地把它写成“团队开发圣经”：

- 目录说明
- 代码规范
- Git 规范
- 测试规范
- 发布规范
- 安全规范
- 文档规范
- UI 规范
- review 规范

最后几百行堆在一起，人都不想看，更别说智能体。

一个真正有效的 `AGENTS.md`，更像：

> **仓库入口地图 + 关键约束提示 + 下一步阅读指引**

它解决的不是“把所有知识塞进去”，而是：

- 让智能体知道先看哪里
- 让它少犯那些最昂贵的错
- 让它知道哪些约束是承重墙
- 让它在需要时跳到更细文档

所以，中小团队写 `AGENTS.md` 时，最重要的不是完整性，而是：

**短、清楚、稳定、可导航。**

---

## 二、一个好的 AGENTS.md 至少要回答 5 个问题

### 1. 这个仓库怎么导航

例如：

- `app/` 放业务代码
- `tests/` 放测试
- `docs/architecture/` 放架构规则
- `scripts/` 放自动化脚本

### 2. 当前任务应先看什么

例如：

- 改 API 先看 `docs/api.md`
- 改前端先看 `docs/ui-guidelines.md`
- 改数据库先看 `docs/schema.md`

### 3. 哪些约束不能越过

例如：

- 不允许跨模块直接访问内部实现
- 不允许跳过 service 层直连数据库
- 不允许引入新状态管理库

### 4. 什么验证是必须通过的

例如：

- `npm test`
- `npm run lint`
- `mvn test`
- `pytest`

### 5. 哪些行为需要停下来确认

例如：

- 删除大量文件
- 改数据库 schema
- 修改基础设施配置
- 改动鉴权与支付链路

如果一份 `AGENTS.md` 没有回答这五个问题，那它大概率只是“项目说明”，还不是高质量智能体入口文件。

## 三、最小模板

下面是一份适合中小团队起步的最小模板。重点不是照抄，而是理解它的结构。

```md
# AGENTS.md

## Project Map

- `src/`：核心业务代码
- `tests/`：单元测试与集成测试
- `docs/architecture/`：架构与边界约束
- `scripts/`：自动化脚本

Start here:

- 改业务逻辑，先看 `docs/architecture/overview.md`
- 改接口，先看 `docs/api-contracts.md`
- 改前端，先看 `docs/ui-guidelines.md`

## Working Rules

- 先复用现有模式，再新增模式
- 小步修改，避免无关重构
- 不要修改与当前任务无关的文件
- 不要猜测需求；不确定时优先查看仓库文档和现有测试

## Hard Constraints

- 不要跨模块访问内部实现
- 不要绕过 service 层直接访问数据库
- 不要引入新的基础框架或状态管理方案
- 不要修改 CI、部署和密钥相关配置，除非任务明确要求

## Verification

- 提交前必须运行：
- `npm run lint`
- `npm test`

如果改动涉及架构边界或公共接口，还要查看：

- `docs/architecture/`
- `docs/api-contracts.md`

## Escalation

以下情况先停下并确认：

- 删除或移动大量文件
- 修改数据库 schema
- 修改鉴权、支付、计费、权限控制
- 修改基础设施、部署、环境变量
```

这份模板故意很短，因为它的任务是“入口导航”，不是把所有细节自己吞下。

## 四、推荐结构：地图在前，规则在中，门控在后

一个顺手的 `AGENTS.md`，推荐按下面顺序组织。

## 1. Project Map

放什么：

- 核心目录职责
- 不同任务先读哪些文件
- 最重要的文档入口

为什么放最前：

- 智能体首先需要知道去哪儿找信息
- 比起抽象原则，导航通常是第一需求

## 2. Working Rules

放什么：

- 默认工作方式
- 编辑策略
- 复用优先
- 小步修改

为什么放第二：

- 这些规则决定“怎么动手”
- 它们比具体业务知识更通用

## 3. Hard Constraints

放什么：

- 架构边界
- 禁止行为
- 高风险区域

为什么单独列：

- 这类规则是承重墙
- 需要比普通建议更醒目

## 4. Verification

放什么：

- 必跑命令
- 必看测试
- 特殊变更的额外检查

为什么要独立成段：

- “完成”不能只靠口头判断
- 验证规则必须显式

## 5. Escalation

放什么：

- 哪些事不能默认做
- 哪些改动必须停下确认

为什么重要：

- 它是风险控制层
- 能防止智能体在高风险区域过度自主

## 五、什么该放进 AGENTS.md，什么不该放

## 建议放进来的

- 仓库导航
- 最重要的项目规则
- 最常犯且代价高的错误提醒
- 关键文档入口
- 最小验证清单
- 高风险变更提醒

## 不建议直接塞进来的

- 全量代码规范细节
- 冗长的框架教程
- 一整套业务背景故事
- 大量低频规则
- 太长的示例代码

这些内容更适合放到：

- `docs/`
- `architecture/`
- `decisions/`
- 专项模板文件

然后在 `AGENTS.md` 里做链接式导航。

## 六、中小团队最常见的 6 个错误

### 错误 1：写得太长

后果：

- 智能体不容易稳定抓到重点
- 人类也不会维护
- 重要规则被淹没

### 错误 2：只写原则，不写入口

后果：

- 智能体知道“要遵守架构”，但不知道去哪里看架构
- 变成空泛口号

### 错误 3：只有建议，没有硬约束

后果：

- 所有规则看起来都一样重
- 智能体不知道哪些不能碰

### 错误 4：没有验证清单

后果：

- “完成”只能靠模型主观判断
- harness 变成说明书，不是控制系统

### 错误 5：把所有知识都放进一个文件

后果：

- 文件变成百科全书
- 失去渐进式披露优势

### 错误 6：写完不更新

后果：

- 文档迅速失真
- 旧规则误导新任务

## 七、一个更实用的演进路径

不要一开始就追求完美版本。

更现实的方式是这样演进：

### 第 1 版：最小地图

包含：

- 仓库导航
- 默认工作方式
- 最小验证命令
- 高风险提醒

### 第 2 版：补关键约束

新增：

- 模块边界
- 常见误区
- 特定任务阅读入口

### 第 3 版：和自动化联动

新增：

- 指向 lint、hooks、结构测试的说明
- 把高频规则从文字升级到工具

### 第 4 版：进入团队飞轮

新增：

- 把复发错误写回 AGENTS 或下级文档
- 建立定期回顾和瘦身机制

这条路径的关键在于：

> **AGENTS.md 不是先规划完再上线，而是边用边长出来。**

## 八、一个适合中小团队的成品示例

下面是一份更接近真实项目的示意稿。

```md
# AGENTS.md

## Project Map

- `backend/`：Spring Boot 服务
- `frontend/`：React 前端
- `docs/architecture/`：架构规则与模块边界
- `docs/domain/`：领域术语与业务规则
- `scripts/`：开发与验证脚本

Start here:

- 改后端接口：先看 `docs/architecture/backend.md`
- 改前端页面：先看 `docs/architecture/frontend.md`
- 改领域逻辑：先看 `docs/domain/glossary.md`

## Working Rules

- 优先复用已有模式，不新增平行实现
- 只修改与当前任务直接相关的文件
- 小步提交，避免顺手重构
- 有测试就先读测试

## Hard Constraints

- 前端不得直接调用数据库或绕过后端 API
- controller 不承载业务规则，业务逻辑放入 service
- 不新增新的 UI 组件库、状态管理库、ORM 框架
- 不修改部署、密钥、鉴权配置，除非任务明确要求

## Verification

后端改动至少运行：

- `mvn test`

前端改动至少运行：

- `npm run lint`
- `npm test`

跨前后端接口改动，还要检查：

- `docs/domain/`
- `docs/architecture/`

## Escalation

以下情况先确认：

- 删除或移动大量文件
- 修改数据库 schema
- 修改支付、权限、鉴权链路
- 修改 CI/CD、环境变量、部署脚本
```

这个版本已经足够让多数编码智能体在一个中小团队仓库里“少走很多弯路”。

## 九、配套建议：别让 AGENTS.md 单兵作战

AGENTS.md 很重要，但它不应该独自承担全部治理责任。

更合理的配套是：

- `AGENTS.md` 负责入口导航
- `docs/architecture/` 负责详细架构规则
- `DECISIONS.md` 或 ADR 负责设计决策
- lint / hooks / tests 负责机械化执行
- review 负责高风险判断

也就是说：

> **AGENTS.md 是入口，不是全部。**

---

## 十、最后的判断标准：怎么知道你的 AGENTS.md 合不合格

你可以用下面 5 个问题自查：

1. 新来的智能体能否知道先看哪几个文件？
2. 它能否区分“建议”与“硬约束”？
3. 它能否知道哪些验证必须跑？
4. 它能否知道哪些变更需要停下确认？
5. 这份文件三个月后团队还愿不愿意维护？

如果前四个问题不能答“能”，它不够有用。  
如果第五个问题不能答“愿意”，它不可持续。

---

## 总结

中小团队写 `AGENTS.md`，最容易走偏的方向，是把它写成“大而全的知识库”。  
但真正高质量的 `AGENTS.md`，更像一张路标清楚的地图：

- 告诉智能体先去哪里
- 告诉它哪些不能碰
- 告诉它什么算完成
- 告诉它什么时候该停下来问

做到这几点，它就已经不是普通说明文档，而是 Harness Engineering 的第一层入口。
