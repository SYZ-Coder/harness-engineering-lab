# Spring 微服务契约测试与可观测性 Playbook

> 面向这样一种现实问题：  
> 微服务系统最大的风险，不是单个服务“代码写错”，而是跨服务兼容性和运行时行为失控。  
> 所以如果想把 Harness Engineering 接进 Spring Cloud / Java 微服务体系，最先值得补的，往往不是更复杂的智能体，而是契约测试与可观测性。

这篇 Playbook 的目标，是讲清楚：

- 为什么这两者应该一起上
- 分别解决什么问题
- 在 Spring 微服务里怎么逐步接入

---

## 一、先说结论：契约测试解决“改不敢改”，可观测性解决“出事看不清”

对重型微服务来说，最常见的两类高成本问题是：

## 问题 1：接口兼容性风险

表现为：

- 一个服务改了 DTO 或字段语义
- 本地能过，联调才炸
- Provider 觉得没问题，Consumer 已经不兼容

这类问题本质上是：

> **跨服务契约没有被前移验证。**

## 问题 2：运行时链路不透明

表现为：

- 某条请求变慢，但不知道慢在哪个服务
- 某个错误是级联触发，但定位要翻很多日志
- 改动上线后，出了问题只能靠经验猜

这类问题本质上是：

> **系统行为没有被统一观察。**

所以，从 Harness Engineering 的视角看：

- 契约测试让系统“可验证”
- 可观测性让系统“可诊断”

这两者结合起来，才真正给智能体和团队提供了足够强的反馈信号。

---

## 二、为什么这两个组合特别适合微服务 Harness

在单体系统里，很多问题可以靠：

- 编译
- 单元测试
- 本地调试

解决掉。

但在微服务里，这远远不够。

原因是：

- 系统行为跨多个服务展开
- 接口变化不一定在当前仓库暴露
- 日志和错误被分散在不同节点

这就意味着：

> 如果没有契约测试，智能体很难知道自己是否破坏了别人的依赖。  
> 如果没有可观测性，智能体也很难知道系统运行时到底发生了什么。

所以这两个能力，几乎是微服务 Harness 最值得优先补的“传感器层”。

---

## 三、契约测试部分：怎么落地

## 1. 契约测试要解决什么

它不是要替代全部集成测试，而是专门解决：

- Provider 变更是否仍兼容 Consumer
- 消费方期待和提供方实现是否一致
- 接口演进是否破坏现有依赖

对 Spring 微服务特别重要的契约对象包括：

- REST API
- Feign / HTTP client 调用
- Kafka / RabbitMQ 事件 payload
- 公共 DTO 或 response schema

## 2. 在 Spring 体系里的常见组合

### 方案 A：Spring Cloud Contract

适合：

- Spring 体系较统一
- 希望直接融入 Spring 测试生态

优势：

- 和 Spring Boot 项目整合自然
- 适合 HTTP 和消息契约

### 方案 B：Pact

适合：

- 多语言服务协作
- Consumer-driven 思路更明显

优势：

- 更强调消费方视角
- 对跨语言微服务更友好

### 方案 C：Microcks

适合：

- 已经有 OpenAPI / AsyncAPI
- 希望集中做 mock、测试、契约治理

优势：

- 更像平台化契约测试入口
- 对规范驱动团队更友好

## 3. 推荐落地顺序

### 第一步：先识别一条最痛调用链

例如：

- `order-service -> payment-service`
- `gateway -> customer-service`
- `inventory-service -> stock events`

### 第二步：把接口事实来源固定下来

至少要明确：

- 规范文件在哪
- provider 谁维护
- consumer 谁依赖

### 第三步：先接一组 provider verification

不要一上来全系统铺开。  
先把最痛的一条链路接进 CI。

### 第四步：把接口变更纳入升级条件

在 `AGENTS.md` 或 PR 规则里明确：

- 改 OpenAPI / DTO / event payload 要补契约验证

## 四、可观测性部分：怎么落地

## 1. 可观测性要解决什么

它不是“多打点”这么简单，而是要回答：

- 请求经过了哪些服务
- 哪一步最慢
- 哪个错误首先出现
- 这次变更是否引入了行为级回归

如果这些问题系统本身答不上来，AI 也答不上来。

## 2. 在 Spring 体系里的常见组合

### 最小组合

- OpenTelemetry Java Agent
- Trace id 贯通
- 日志关联 trace id

### 进阶组合

- Spring Boot Starter
- 自定义 span / attribute
- 关键业务指标埋点

### 平台组合

- OTel Collector
- Jaeger / Tempo / Grafana / ELK 等后端
- 日志、指标、链路统一查询入口

## 3. 推荐落地顺序

### 第一步：先打通 trace

先不要一开始就追求很细的业务语义埋点。  
先做到：

- 请求可跨服务串起来
- 日志能带 trace id

### 第二步：再补关键服务的业务指标

例如：

- 下单成功率
- 支付确认耗时
- 库存校验失败率

### 第三步：再把诊断入口标准化

比如：

- 某条链路在哪里查
- 出错时先看什么 dashboard
- 哪些日志关键字最重要

这一步很关键，因为 Harness 不只是“有数据”，还要“能用数据”。

---

## 五、为什么契约测试和可观测性最好一起上

因为它们分别覆盖了两个不同阶段：

## 契约测试：上线前信号

作用：

- 在变更进入系统前，发现兼容性问题

## 可观测性：上线后信号

作用：

- 在真实流量和真实链路中，发现行为级问题

把两者合起来，你会得到一个更完整的闭环：

1. 改代码前知道哪些契约不能破
2. 改代码时有契约验证做门控
3. 改代码后能从 trace / log / metric 观察真实结果

这才是适合微服务的 Harness 反馈层。

---

## 六、一个适合 Spring 微服务团队的最小 Playbook

如果你的团队现在就要做，我建议这样起步。

## 第 1 周：契约试点

- 选一条最痛的服务调用链
- 选定 Spring Cloud Contract / Pact / Microcks 之一
- 把 provider verification 接进 CI

## 第 2 周：可观测性试点

- 给 2 到 3 个核心服务接 OTel
- 至少打通 trace id
- 让日志和 trace 可关联

## 第 3 周：把规则接进入口文件

- 在各服务 `AGENTS.md` 中写清：
  - 改接口需补契约验证
  - 改跨服务调用需关注可观测信号

## 第 4 周：做第一次复盘

- 哪些兼容性问题被提前拦住了
- 哪些运行时问题仍然只能靠人工经验定位
- 哪些规则还没写清楚

## 七、与 AI / Harness Engineering 的结合方式

很多团队会问：

> 这些不是常规工程治理吗？和 Harness Engineering 有什么关系？

关系就在于：

- 契约测试提供了机器可执行的“兼容性边界”
- 可观测性提供了运行时真实反馈信号
- 智能体因此不再只面对静态代码，而是面对一个更完整的工程环境

具体结合方式可以是：

### 1. 改接口时

智能体先看：

- OpenAPI / 契约测试
- 当前 consumer 约束

### 2. 改跨服务逻辑时

智能体不仅改代码，还要：

- 运行契约相关验证
- 查看 trace / log 信号

### 3. 做问题定位时

智能体先读：

- 错误日志
- trace 链路
- 关键指标变化

这时候，AI 的角色就从“会写代码”变成了“能在规则和信号中工作”。

---

## 八、最容易踩的坑

## 坑 1：只有 OpenAPI，没有验证

结果：

- 规范写了，但没人真正验证兼容性

## 坑 2：只有 trace，没有查询入口

结果：

- 数据有了，但团队和智能体都不会用

## 坑 3：只做 provider 视角，不看 consumer 影响

结果：

- “我这边没问题”依然会变成联调事故

## 坑 4：可观测性只做平台，不接到服务变更流程

结果：

- 它只是监控系统，不是 Harness 反馈层

## 坑 5：试图一次覆盖所有服务

结果：

- 复杂度和阻力同时飙升

---

## 九、总结

对于 Spring Cloud / Java 微服务来说，契约测试和可观测性之所以重要，不是因为它们“先进”，而是因为它们正好补在微服务最痛的两个地方：

- 契约测试补“改不敢改”
- 可观测性补“出事看不清”

从 Harness Engineering 的视角看，它们分别提供了：

- 上线前的兼容性门控
- 上线后的行为反馈信号

这两者一起接入后，微服务系统才会真正变成一个：

> **对智能体可验证、对团队可诊断、对演进可治理的工程环境。**
